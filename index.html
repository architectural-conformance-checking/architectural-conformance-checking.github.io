Checagem de Conformidade Arquitetural

Projetando Arquitetura de Software

Descrever o termo arquitetura de software realmente é uma tarefa difícil. Podemos encontrar várias definições que falam de componentes arquiteturais, relacionamentos, definições organizacionais, distribuição de sistemas e várias outras coisas que falam de como organizar o sistema para manter um alto nível de manutenibilidade.

No início do desenvolvimento de um sistema, em geral, há uma etapa de especificação da arquitetura a fim de estabelecer os principais componentes, as suas interações e como eles irão funcionar em conjunto para atender os requisitos e respeitar certas restrições.

Esses componentes que compoẽm a arquitetura podem ser camadas internas que foram projetadas com o objetivo de organizar o código e manter a manutenibilidade ao longo do tempo. Por exemplo, imagine um sistema web para agendamento de consultas médicas e que, internamente, foi organizado utilizando camadas. Uma camada de apresentação (Controllers) responsável por controlar o fluxo de requisições, uma camada de serviço (Services) responsável por executar a lógica de negócio e uma camada chamada Repositories responsável por fazer acesso a dados. As camadas desse sistema podem ser visualizadas na figura abaixo.

Perceba que até aqui, apenas camadas e suas responsabilidades foram projetadas. O próximo passo é projetar as interações entre essas camadas. No caso do sistema de agendamento de consultas, as camadas superiores fazem chamadas para as camadas inferiores. Isso é, a camada de Controllers faz chamadas para a camada de Services e essa faz chamadas para a camadas de Repositories. Note que essas dependências são representadas pelas setas entre as camadas na imagem a seguir.

Como dito anteriormente, projetar arquitetura diz a respeito de estabelecer componentes arquiteturais e como eles interagem a fim de atender requisitos e respeitar restrições. Essas restrições são, por exemplo, dizer que a camadas de controllers NÃO pode fazer chamadas diretas para a camadas de Repositories, bem como a camadas de Services não pode fazer chamadas para a camadas de Controllers. Caso essas chamadas aconteçam, (as chamadas que não deveriam acontecer) elas se caracterizam como Desvios Arquiteturais uma vez que a implementação se desvia da arquitetura que foi planejada inicialmente.

Como Identificar Desvio Arquiteturais

Esses desvios arquiteturais acabam degradando o sistema com o passar do tempo. Dessa forma, precisamos de alguma abordagem que consiga comparar a arquitetura planejada com a implementação e revelar esses desvios arquitetônicos.

Em geral, essas abordagens tomam como entrada uma descrição da arquitetura que foi planejada e a implementação real e comparam as duas para verificar se a implementação está de acordo com o que foi planejado - como mostrado na imagem abaixo.

De forma geral, é assim que a checagem de conformidade arquitetural funciona. Vale ressaltar que a Arquitetura Planejada é especificada em uma linguagem de domínio, permitindo que esse processo seja totalmente automatizado.
